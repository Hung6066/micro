# Spec: His.Hope Agent Knowledge Base

> **Status**: Draft | **Author**: @architect | **Date**: 2026-07-17

## 1. Problem Statement

His.Hope có 16 AI agents chuyên biệt, mỗi lần làm việc phải đọc lại toàn bộ project để hiểu ngữ cảnh. Các bài học kinh nghiệm (gotchas, patterns, architectural decisions) hiện nằm rải rác trong ADRs, coding-standards, hoặc chỉ tồn tại trong đầu developer. Agents không có cơ chế tra cứu nhanh các "bài học đã biết" để tránh lặp lại lỗi cũ.

**Cần**: Một kho kiến thức có cấu trúc, agents tra cứu được theo tag, và có cơ chế đóng góp từ cả người lẫn agents.

---

## 2. Design Decisions

| Decision | Rationale |
|---|---|
| **Structured Markdown + YAML frontmatter** | Đủ cấu trúc để parse tự động, vẫn dễ đọc bằng mắt. Không cần tooling phức tạp. |
| **1 entry = 1 file** | Dễ review riêng, không merge conflict, dễ move giữa `.capture/` và `entries/`. |
| **Tổ chức theo domain** | Khớp với cấu trúc agent hiện tại. Mỗi domain có thư mục riêng. |
| **Tag-based lookup** | Tiết kiệm context. Agent chỉ đọc entry liên quan đến task, không phải toàn bộ kho. |
| **`.capture/` staging area** | Tách biệt entry đã duyệt và chưa duyệt. Human review gate trước khi vào kho chính. |
| **INDEX.md auto-generate** | Đảm bảo index luôn khớp với entries, không bị stale thủ công. |

**Naming convention**: `{domain}-{type}-{nn}-{slug}.md`
- `domain`: short domain name (patient, frontend, devops, db, ...)
- `type`: `gotcha` | `pattern` | `decision`
- `nn`: sequential number, zero-padded (01, 02, ...)
- `slug`: kebab-case short description

## 4. Entry Format

Every entry is a Markdown file with YAML frontmatter between `---` delimiters.

### 4.1 Frontmatter Schema

| Field | Required | Values | Description |
|---|---|---|---|
| `id` | Yes | `{domain}-{type}-{nn}` | Unique ID matching filename |
| `type` | Yes | `gotcha` / `pattern` / `decision` | Knowledge type |
| `domain` | Yes | domain directory name | Owning domain |
| `tags` | Yes | `[tag1, tag2, ...]` | Lookup keywords (min 2) |
| `severity` | Yes | `critical` / `warning` / `info` | Importance level |
| `agent` | Yes | `@agent-name` | Responsible agent for this domain |
| `author` | Yes | `@agent-name` or human name | Entry creator |
| `date` | Yes | `YYYY-MM-DD` | Creation date |
| `related` | No | `[id1, id2, ...]` | Related entries or ADRs |

### 4.2 Entry Types

**Gotcha** (`type: gotcha`): Lỗi đã từng xảy ra, cách tránh, cách phát hiện.
- Sections: Problem, Consequence, Detection, Correct Approach, Where It Happened

**Pattern** (`type: pattern`): Mẫu code chuẩn đã được chứng minh.
- Sections: When To Use, Standard Template, Rationale, References

**Decision** (`type: decision`): Quyết định kiến trúc, trade-off đã cân nhắc.
- Sections: Decision, Rationale, Trade-offs, When To Revisit, References

### 4.3 Standard Tags

| Tag Group | Examples |
|---|---|
| **Tech stack** | `dotnet`, `angular`, `cockroachdb`, `kubernetes`, `linkerd`, `redis`, `rabbitmq`, `vault`, `docker`, `grpc`, `rest`, `bazel`, `tekton`, `argocd` |
| **Pattern/Concept** | `aggregate`, `cqrs`, `outbox`, `saga`, `circuit-breaker`, `ngrx`, `onpush`, `lazy-loading`, `rls`, `migration`, `mTLS` |
| **Problem** | `deadlock`, `race-condition`, `n+1`, `memory-leak`, `xss`, `sql-injection`, `secrets`, `timeout`, `concurrency`, `performance` |

Every entry must have at least 1 tech tag + 1 problem/concept tag.

## 6. Agent Workflows

### 6.1 Consumption (Reading Knowledge)

```
Agent receives task from @architect/@dispatcher
  → Read INDEX.md (~50 lines, low context cost)
  → Identify relevant tags from task description
  → If matching entries exist: read them (typically 1-3 files)
  → Apply knowledge, avoid known mistakes
  → Complete task
```

**Rule**: Agent MUST consult INDEX.md before coding when task involves:
- Bug fix → search for gotchas in same domain
- New feature → search for patterns in same domain
- Architecture change → search for related decisions

### 6.2 Contribution (Capturing Knowledge)

```
Agent completes task (bug fix / feature)
  → Self-assess: "Is there a lesson worth saving?"
    ✓ Bug caused by a known but undocumented pattern
    ✓ Solution non-obvious, needs explanation
    ✓ Bug took >30 min to debug
    ✓ Decision affecting multiple services
  → If yes: create entry in .capture/
  → Notify @architect: "N pending knowledge entries"
  → Human review:
    ├── APPROVE → Move to entries/, run generate-index.ps1
    ├── EDIT    → Revise content, then approve
    └── REJECT  → Delete or leave with reason
```

**Agent rules for creating entries**:
- 1 entry = 1 lesson (don't combine multiple issues)
- Always include code examples (correct + incorrect)
- Always state consequences (what happens if ignored)
- Minimum 2 tags (1 tech + 1 problem)
- Check INDEX.md first to avoid duplicates

### 6.3 `.capture/` Entry Format

Same as standard entry, plus two extra frontmatter fields:

```yaml
status: pending-review
reviewer: @architect
```

## 7. Pipeline Integration

### 7.1 Integration Points in 5-Phase Pipeline

| Phase | New Behavior |
|---|---|
| **Phase 0** | Agent reads INDEX.md, loads relevant gotchas. @dispatcher suggests tags. |
| **Phase 2** | Agent follows known patterns from knowledge base. Notes non-obvious bugs for later capture. |
| **Phase 4** | @validate checks: index freshness, entry validity, stale captures. @docs handles knowledge-capture. |
| **Phase 5** | @git warns if `.capture/` has unreviewed entries. |

### 7.2 New Quality Gates

| Gate ID | Phase | Severity | Agent | Description |
|---|---|---|---|---|
| `knowledge-index-fresh` | 4 | block | @validate | INDEX.md entry count matches `entries/` file count |
| `knowledge-entries-valid` | 4 | block | @validate | All entries have valid frontmatter, non-empty tags |
| `knowledge-no-stale-capture` | 4 | warn | @validate | Warn if `.capture/` has files >7 days unreviewed |

## 8. opencode.json Changes

### 8.1 New Project Reference

```json
"knowledge": {
  "path": "docs/knowledge",
  "description": "Use for agent knowledge base: gotchas, patterns, decisions"
}
```

### 8.2 Updated Agent Descriptions

**@docs** — adds "knowledge base (capture & verify)":

> Documentation agent — ADRs, API docs, service READMEs, changelogs, runbooks, knowledge base (capture & verify). Uses MCP: filesystem for reading/writing docs, github for git operations on doc files.

**@validate** — adds "knowledge index freshness":

> Validation agent (API contract, schema, FluentValidation, build, config/secrets, migration safety, knowledge index freshness). Uses MCP: db-* for migration verification, docker for container validation, filesystem for config inspection.

**@explore** — adds knowledge base lookup role (implicit, no config change needed).

### 8.3 Agent Responsibilities

| Agent | Knowledge Base Responsibility |
|---|---|
| @architect | Review & approve/reject entries from `.capture/` |
| @dispatcher | Suggest relevant tags during task analysis |
| @docs | Generate new entries (Phase 2), verify index (Phase 4) |
| @explore | Look up INDEX.md + read entries on behalf of other agents |
| @validate | Gate: `knowledge-index-fresh`, `knowledge-entries-valid`, `knowledge-no-stale-capture` |
| @git | Warn if `.capture/` has pending entries before commit |
| All implementation agents | Read INDEX.md before coding; propose entries to `.capture/` after non-obvious fixes |

## 9. Script: generate-index.ps1

Located at `docs/knowledge/scripts/generate-index.ps1`.

Scans all `.md` files under `entries/`, parses YAML frontmatter, generates `INDEX.md` with three tables (by tag, by domain, by severity).

Run manually after adding/moving entries, or via pre-commit hook (optional).
